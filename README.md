# Nest Marketplace — منصة تجارة إلكترونية متعددة البائعين

نسخة MVP قابلة للتشغيل محليًا لمنصة شبيهة بفكرة Amazon وNoon، وليست متجرًا لبائع واحد.

## ما الذي يعمل الآن؟

### مالك المنصة (Super Admin)
- لوحة مؤشرات للمبيعات والطلبات والعملاء والبائعين.
- قبول ورفض وإيقاف المتاجر.
- تحديد عمولة افتراضية للمنصة أو عمولة خاصة لكل متجر.
- مراجعة المنتجات قبل ظهورها للعملاء.
- مراجعة تقييمات العملاء.
- إنشاء الأقسام.
- متابعة كل الطلبات وعمولة المنصة لكل طلب.
- مراجعة طلبات سحب أرباح البائعين واعتمادها أو رفضها أو تعليمها كمدفوعة.
- تغيير اسم المنصة والعملة وبريد الدعم ونسبة العمولة.

### البائع
- إنشاء حساب متجر ينتظر موافقة مالك المنصة.
- لوحة مستقلة لكل متجر.
- إضافة وتعديل المنتجات والمخزون والأسعار والصور.
- رفع صورة المنتج من الجهاز أو استخدام رابط/اسم صورة.
- متابعة الطلبات الخاصة بمتجره فقط.
- تحديث حالة الطلب: تجهيز، شحن، تسليم، أو إلغاء.
- احتساب العمولة وصافي مستحقات البائع تلقائيًا.
- طلب سحب الأرباح المتاحة.

### العميل
- إنشاء حساب وتسجيل الدخول.
- تصفح الأقسام والمتاجر والمنتجات.
- شراء منتجات من أكثر من متجر في سلة واحدة.
- تقسيم الطلب تلقائيًا إلى طلب فرعي مستقل لكل بائع.
- متابعة الطلبات السابقة.
- إرسال تقييم لمنتج سبق شراؤه، بعد مراجعة الإدارة.

## التشغيل

المتطلبات: Node.js 22 أو أحدث.

```bash
npm start
```

ثم افتح:

- الموقع: `http://localhost:3000`
- لوحة المالك: `http://localhost:3000/owner.html`
- لوحة البائع: `http://localhost:3000/vendor-dashboard.html`

لا تحتاج إلى تثبيت MySQL. قاعدة SQLite تُنشأ تلقائيًا باسم `marketplace.db` عند أول تشغيل.

## حسابات التجربة

### مالك المنصة
- البريد: `owner@nest.local`
- كلمة المرور: `Owner123!`

### بائع المنتجات الغذائية
- البريد: `vendor@fresh.local`
- كلمة المرور: `Vendor123!`

### بائع الأجهزة
- البريد: `vendor@tech.local`
- كلمة المرور: `Vendor123!`

### عميل
- البريد: `customer@demo.local`
- كلمة المرور: `Customer123!`

غيّر بيانات الدخول الافتراضية قبل نشر المشروع.

## كيف تُحسب الأرباح؟

عند إنشاء طلب من عدة متاجر، ينشئ النظام:
1. طلبًا رئيسيًا للعميل.
2. طلبًا فرعيًا لكل متجر.
3. قيمة عمولة المنصة لكل طلب فرعي.
4. صافي مستحقات البائع بعد خصم العمولة.

الرصيد يصبح متاحًا للسحب بعد تعليم البائع للطلب بأنه تم تسليمه.

## إعادة قاعدة البيانات للوضع التجريبي

أوقف الخادم، احذف ملفات:
- `marketplace.db`
- `marketplace.db-wal`
- `marketplace.db-shm`

ثم شغّل `npm start` من جديد.

## مهم قبل الإطلاق التجاري الحقيقي

هذه نسخة MVP قوية للتجربة والتطوير، وليست إعدادًا نهائيًا للإنتاج. قبل استقبال أموال حقيقية يجب إضافة:
- بوابة دفع فعلية مثل Paymob أو Fawry أو Stripe بمفاتيح حساب التاجر.
- تخزين الصور على خدمة Object Storage بدل حفظ الصور المرفوعة داخل SQLite.
- HTTPS ودومين واستضافة وإعداد نسخ احتياطي.
- إرسال بريد ورسائل تحقق واستعادة كلمة المرور.
- سياسة استرجاع وشحن وضرائب وفواتير مناسبة للدولة.
- اختبارات أمنية ومراجعة قانونية لشروط الاستخدام والخصوصية.
- نقل قاعدة البيانات إلى PostgreSQL أو MySQL عند زيادة عدد المستخدمين والطلبات.

راجع `ARCHITECTURE.md` و`PRODUCTION_CHECKLIST.md` لمزيد من التفاصيل.

## النشر على الإنترنت

هذه النسخة تتضمن `render.yaml` وتهيئة قاعدة بيانات مستمرة وملفات `robots.txt` و`sitemap.xml` ديناميكية. راجع `DEPLOY_ONLINE_AR.md` للخطوات الكاملة.

في وضع الإنتاج لن يعمل المشروع أول مرة بدون ضبط `OWNER_EMAIL` و`OWNER_PASSWORD`، ولن تُستخدم كلمة مرور المالك التجريبية المعروفة.
